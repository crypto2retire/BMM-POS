STEP 2 — Handle tool calls in the assistant endpoint

After calling OpenRouter, check if the response contains tool_calls. If it does:

1. Parse the tool name and arguments from the tool call
2. Execute the corresponding internal API action using the vendor's auth token
3. Format the result into a human-readable confirmation message
4. Make a second OpenRouter call with the tool result included so the assistant can respond naturally

Tool execution logic (run internally using SQLAlchemy, not HTTP):

For add_item:
- Set vendor_id from the authenticated user's ID
- Generate SKU and barcode using the existing barcode service
- Insert into items table
- Return: "I added [name] to your inventory! The SKU is [sku] and the price is $[price]. Would you like me to set it as available online too?"

For edit_item:
- Verify the item belongs to the authenticated vendor
- Update only the fields provided
- Return: "Done! I updated [item name]. [summary of what changed]"

For archive_item:
- Verify the item belongs to the authenticated vendor
- Set status to 'removed'
- Return: "I archived [item name]. It won't show in the POS or online anymore. You can restore it anytime."

For list_items:
- Query items for this vendor with filters applied
- Return a compact list: ID, name, price, status
- Format as a numbered list the assistant can reference

For get_item:
- If item_id provided: look up directly
- If name provided: search by name ILIKE for this vendor
- Return full item details

STEP 4 — Update the assistant chat schema

Update app/schemas/assistant.py (create it if it doesn't exist):
```python
from pydantic import BaseModel
from typing import Optional

class AssistantChatRequest(BaseModel):
    message: str
    image_base64: Optional[str] = None
    image_mime_type: Optional[str] = None

class AssistantChatResponse(BaseModel):
    reply: str
    action_taken: Optional[str] = None  # e.g. "item_added", "item_edited", "item_archived"
    item_id: Optional[int] = None  # ID of item that was created or modified
```
