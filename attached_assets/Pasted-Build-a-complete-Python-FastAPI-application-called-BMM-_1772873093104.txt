Build a complete Python FastAPI application called BMM-POS (Bowenstreet Market POS system). This is a point-of-sale and vendor management system for a vendor mall with 120 vendors.

Use the built-in Replit PostgreSQL database. Use the DATABASE_URL environment variable that Replit provides automatically.

TECH STACK:
- Python FastAPI backend
- PostgreSQL database (Replit built-in)
- Plain HTML + vanilla JavaScript frontend (no React, no Vue, no frameworks)
- SQLAlchemy async ORM
- JWT authentication with python-jose and passlib bcrypt

STEP 5 — Create migrations/001_initial_schema.sql with these exact tables:

vendors table:
- id SERIAL PRIMARY KEY
- name VARCHAR(255) NOT NULL
- email VARCHAR(255) UNIQUE NOT NULL
- phone VARCHAR(20)
- booth_number VARCHAR(20)
- monthly_rent NUMERIC(10,2) NOT NULL DEFAULT 0
- rent_due_day INTEGER NOT NULL DEFAULT 27
- password_hash VARCHAR(255) NOT NULL
- role VARCHAR(20) NOT NULL DEFAULT 'vendor' (values: vendor, admin)
- payout_method VARCHAR(20) DEFAULT 'zelle' (values: zelle, check, cash)
- zelle_handle VARCHAR(255)
- status VARCHAR(20) NOT NULL DEFAULT 'active' (values: active, suspended)
- created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()

vendor_balances table:
- id SERIAL PRIMARY KEY
- vendor_id INTEGER UNIQUE NOT NULL REFERENCES vendors(id) ON DELETE CASCADE
- balance NUMERIC(10,2) NOT NULL DEFAULT 0
- last_updated TIMESTAMPTZ NOT NULL DEFAULT NOW()

items table:
- id SERIAL PRIMARY KEY
- vendor_id INTEGER NOT NULL REFERENCES vendors(id) ON DELETE CASCADE
- sku VARCHAR(100) UNIQUE NOT NULL
- barcode VARCHAR(100) UNIQUE NOT NULL
- name VARCHAR(255) NOT NULL
- description TEXT
- category VARCHAR(100)
- price NUMERIC(10,2) NOT NULL
- quantity INTEGER NOT NULL DEFAULT 1
- photo_urls TEXT[]
- is_online BOOLEAN NOT NULL DEFAULT FALSE
- is_tax_exempt BOOLEAN NOT NULL DEFAULT FALSE
- sale_price NUMERIC(10,2)
- sale_start DATE
- sale_end DATE
- status VARCHAR(20) NOT NULL DEFAULT 'active' (values: active, sold, removed)
- created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()

sales table:
- id SERIAL PRIMARY KEY
- cashier_id INTEGER REFERENCES vendors(id)
- subtotal NUMERIC(10,2) NOT NULL
- tax_rate NUMERIC(5,4) NOT NULL DEFAULT 0.0550
- tax_amount NUMERIC(10,2) NOT NULL
- total NUMERIC(10,2) NOT NULL
- payment_method VARCHAR(20) NOT NULL (values: cash, card, split)
- cash_tendered NUMERIC(10,2)
- change_given NUMERIC(10,2)
- card_transaction_id VARCHAR(255)
- receipt_email VARCHAR(255)
- created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()

sale_items table:
- id SERIAL PRIMARY KEY
- sale_id INTEGER NOT NULL REFERENCES sales(id) ON DELETE CASCADE
- item_id INTEGER NOT NULL REFERENCES items(id)
- vendor_id INTEGER NOT NULL REFERENCES vendors(id)
- quantity INTEGER NOT NULL DEFAULT 1
- unit_price NUMERIC(10,2) NOT NULL
- line_total NUMERIC(10,2) NOT NULL

rent_payments table:
- id SERIAL PRIMARY KEY
- vendor_id INTEGER NOT NULL REFERENCES vendors(id)
- amount NUMERIC(10,2) NOT NULL
- period_month DATE NOT NULL
- method VARCHAR(20) NOT NULL DEFAULT 'balance'
- status VARCHAR(20) NOT NULL DEFAULT 'paid' (values: paid, insufficient, waived)
- notes TEXT
- processed_at TIMESTAMPTZ NOT NULL DEFAULT NOW()

payouts table:
- id SERIAL PRIMARY KEY
- vendor_id INTEGER NOT NULL REFERENCES vendors(id)
- period_month DATE NOT NULL
- gross_sales NUMERIC(10,2) NOT NULL DEFAULT 0
- rent_deducted NUMERIC(10,2) NOT NULL DEFAULT 0
- net_payout NUMERIC(10,2) NOT NULL DEFAULT 0
- payout_method VARCHAR(20)
- zelle_handle VARCHAR(255)
- status VARCHAR(20) NOT NULL DEFAULT 'pending' (values: pending, sent, paid)
- paid_at TIMESTAMPTZ
- notes TEXT
- created_at TIMESTAMPTZ NOT NULL DEFAULT NOW()

Add indexes on:
- items(vendor_id), items(barcode), items(status)
- sale_items(sale_id), sale_items(vendor_id)
- sales(created_at)
- rent_payments(vendor_id, period_month)
- payouts(vendor_id, period_month)

Add a PostgreSQL trigger that automatically creates a vendor_balances row whenever a new vendor is inserted.

STEP 10 — Create app/schemas/item.py
Pydantic schemas:
- ItemCreate: all item fields, barcode is optional (for Ricochet imports), vendor_id optional (admin must provide, vendors auto-set from auth)
- ItemUpdate: all fields optional
- ItemResponse: all fields plus computed active_price field (returns sale_price if today is between sale_start and sale_end, otherwise returns regular price), model_config from_attributes=True

STEP 11 — Create app/services/barcode.py
Two functions:
- generate_sku(vendor_id, db) — generates SKU in format BSM-{vendor_id:04d}-{sequence:06d}, checks for collisions
- generate_barcode_image(barcode_value, output_path) — generates Code 128 barcode PNG using python-barcode

STEP 12 — Create app/services/labels.py
Two functions using ReportLab:
- generate_label_pdf(item) — single 2.25 x 1.25 inch Zebra-compatible label showing item name, price, booth number, and Code 128 barcode
- generate_label_sheet(items) — batch PDF with one label per page

STEP 13 — Create app/routers/auth.py
- POST /api/v1/auth/login — OAuth2 form, email + password, returns JWT
- POST /api/v1/auth/logout — stateless, client discards token
- get_current_user dependency — decodes JWT, loads vendor from DB, checks active status
- require_admin dependency — chains on get_current_user, checks role == admin

STEP 14 — Create app/routers/vendors.py
- GET /api/v1/vendors/ — admin only, returns all vendors with balances
- POST /api/v1/vendors/ — admin only, creates vendor, checks for duplicate email
- GET /api/v1/vendors/{id} — admin sees any vendor, vendor sees only themselves
- PUT /api/v1/vendors/{id} — admin can edit all fields, vendors can edit own profile but NOT role, status, monthly_rent, rent_due_day, or booth_number
- DELETE /api/v1/vendors/{id} — admin only, soft delete sets status to suspended

STEP 15 — Create app/routers/items.py
- GET /api/v1/items/ — admin sees all items, vendor sees only their own, filterable by status and category
- POST /api/v1/items/ — creates item, auto-generates SKU and barcode unless barcode is provided (Ricochet import), vendor_id auto-set from auth for vendor role
- GET /api/v1/items/barcode/{barcode} — POS lookup by barcode, returns item with active_price
- GET /api/v1/items/{id} — single item with access control
- PUT /api/v1/items/{id} — vendors can update their own items including setting sale_price, sale_start, sale_end
- DELETE /api/v1/items/{id} — soft delete sets status to removed
- GET /api/v1/items/{id}/label — returns PDF label for that item

STEP 19 — Create frontend/static/js/api.js
Shared JavaScript fetch wrapper:
- Store JWT token in a JavaScript variable in memory only, never localStorage or sessionStorage
- apiLogin(email, password) — posts to /api/v1/auth/login using OAuth2 form encoding, stores token, returns decoded JWT payload
- apiFetch(method, url, body) — internal function that adds Authorization Bearer header automatically
- apiGet(url) — calls apiFetch GET
- apiPost(url, body) — calls apiFetch POST
- apiPut(url, body) — calls apiFetch PUT
- apiDelete(url) — calls apiFetch DELETE
- On 401 response: clear token and redirect to /vendor/login.html
- On 422 response: extract and display FastAPI validation error messages
- getToken() and clearToken() helper functions
- parseToken() — decodes JWT payload without verification for reading role and vendor id
